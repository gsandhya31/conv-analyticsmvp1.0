Product Thinking
Why NL→SQL over a Dashboard?
  • Dashboards answer pre-defined questions
  • NL→SQL answers ad-hoc questions nobody anticipated
  • No dependency on dashboard redesign cycles
  • Reduces "one more chart" requests to engineering
Constraints Defined
  • Read-only: system can never modify data
  • PostgreSQL (Supabase) as data layer
  • Claude Sonnet4 for SQL generation
  • HQ ops team persona — no role-based access needed
User Persona
  • HQ operations managers & analysts
  • Non-technical: can't write SQL
  • Need quick answers for daily decisions
  • Currently depend on data engineering team
Scope Boundaries
  • ✅ Data retrieval & aggregation
  • ✅ Multi-turn follow-up questions
  • ✅ Query logging & analytics
  • ❌ Forecasting / predictions
  • ❌ Data modification
  • ❌ User authentication
02b
AI Prototyping Decisions — Why This Stack
Why Python (not Node/Go)
  • First-class Anthropic SDK support — anthropic pip package
  • Supabase Python client for DB operations
  • Data manipulation with Pandas built in
  • Fastest path from idea → working backend for a solo builder
  • Industry standard for AI/ML prototyping
Why Streamlit (not Lovable/Cursor)
  • Single Python file → full chat UI. No HTML/CSS/JS needed
  • Built-in: st.chat_input, st.dataframe, st.expander — all we need
  • Lovable/Cursor generate React code — adds a separate frontend to maintain
  • Streamlit keeps backend logic + UI in one codebase, one deploy
  • Tradeoff accepted: limited styling control. Fine for an internal ops tool.
Why Supabase (not raw PostgreSQL)
  • Managed PostgreSQL — no server setup, no DevOps
  • Built-in REST API + RPC functions for raw SQL execution
  • Table editor for quick data inspection during development
  • Free tier sufficient for prototype scale (10K rows, low QPS)
  • Tradeoff: RPC function needed for raw SQL — Supabase client doesn't support it natively
Why Render (not Vercel/AWS)
  • Vercel doesn't support Python server apps (Streamlit needs a persistent process)
  • AWS/GCP: overkill setup time for a demo
  • Render: connect GitHub repo → set env vars → deploy. Under 5 minutes.
  • Free tier with auto-deploy on git push
  • Tradeoff accepted: cold start ~2mins after inactivity. Pre-warm before demo.
Why Claude Sonnet 4 (not GPT-4/Gemini)
  • Strong SQL generation accuracy with schema-aware prompting
  • Structured JSON output reliability — fewer parsing failures
  • Competitive pricing: $3/1M input, $15/1M output tokens
  • Clean API — system prompt + messages pattern fits NL→SQL well
  • Builder familiarity — faster iteration when you know the tool
What I'd Change for Production
  • Frontend: React + FastAPI — full control over UI/UX
  • Hosting: AWS ECS ,Azure App Service or GCP Cloud Run — no cold starts, auto-scaling
  • DB: Direct PostgreSQL connection pool — skip RPC overhead
  • Caching: Redis for repeated queries — reduce LLM calls + latency
  • Semantic layer: between LLM and raw tables
  • Observability: LangSmith or Langfuse for LLM tracing

app.py receives user input
  → calls sql_generator.generate_sql()
    → prompts.build_prompt() constructs system prompt + messages array
    → Anthropic API call → Claude Sonnet 4 (max_tokens: 1024)
    → returns JSON: {sql, explanation, assumptions} + token metadata
  → calls db.execute_query() → Supabase RPC (SELECT/WITH only)

Metrics Framework — Business & Product
Business Metrics(targets)
  • Adoption rate: 60%+ of ops team active weekly within 3 months of launch
  • DE bypass rate: 40% reduction in ad-hoc data requests to engineering within 2 months
  • Time to insight: from hours/days (ticket → engineer) to under 10 seconds (self-serve)
  • Query satisfaction: 80%+ thumbs up rate on query outputs
  • Repeat usage: active users averaging 5+ queries per week
Product Metrics(targets)
  • E2E latency: <3 sec for simple queries, <6 sec for complex (CTE/window)
  • SQL accuracy: 90%+ valid SQL on first attempt (no execution errors)
  • Retry rate: <15% of queries need rephrasing to get expected results
  • Cost per query: <₹0.80 average (currently ~₹0.50)
  • Uptime: 99.5%+ availability during business hours (9am–7pm IST)
  • Graceful failure: 100% of unanswerable questions get a clear explanation, not an error
06
Guardrails & Safety
Implemented
Read-Only Enforcement
Only SELECT / WITH queries allowed. Validated at both application layer and database function level.
Implemented
No Forecasting / Extrapolation
LLM is instructed to refuse prediction queries. The system retrieves data, it doesn't generate projections.
Implemented
Full Audit Trail
Every query logged: user question, generated SQL, tokens used, latency, success/failure, full LLM request & response.
Implemented
Schema Isolation
Company data stays in Supabase. Only the schema structure (not actual data) is sent to the LLM for SQL generation.
To Implement
PII / Sensitive Data Redaction
Redact company-sensitive information from user queries before they hit the LLM. Scrub names, IDs, proprietary terms.
To Implement
Query Cost Limits
Cap expensive queries (full table scans, massive joins) to prevent runaway costs on production-scale data.
07
Key Assumptions
  • Users are HQ ops team with no SQL knowledge — they ask in plain English
  • All users see all data — no row-level security or role-based access required
  • 2-table schema is sufficient to demonstrate the pattern; production would have 10-50+ tables
  • Synthetic data (30 stations, 6 months) is structurally representative of real fuel retail operations
  • Claude Sonnet 4 (claude-sonnet-4-20250514) via Anthropic API is accurate enough for SQL generation on well-defined schemas with few-shot prompting
  • Multi-turn context (last 5 turns) is sufficient for follow-up questions without full conversation replay
  • Cold start latency on free Render tier (~2mins) is acceptable for a demo / interview prototype
  • Date references ("last month", "this quarter") are anchored to the data range, not real-world calendar
